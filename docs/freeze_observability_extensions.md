## Daily checklist (10 minutes, after 16:00 IST)

The operator's only routine task during the freeze. Skip more than
two days in a row and the freeze is in §C6 (operator drift)
territory.

1. Open `logs/diagnostics/eod_YYYY-MM-DD.md` (auto-generated by the
   trader's EOD hook around 15:35 IST).
2. Skim the **top section only** — portfolio PF and trade count.
   Do not read the per-strategy section yet on bad days; it biases.
3. Open `docs/freeze_log_week<N>.md` (per-week running notes,
   created Friday → next Friday). Append two lines:
   - `YYYY-MM-DD | longs N=__ PnL=Rs __ | shorts N=__ PnL=Rs __`
   - `YYYY-MM-DD | audit: GREEN / AMBER / RED — <one-line reason>`
4. If any strategy hit its per-day kill (3 consec losses or −1 % cap),
   note it. **Do not disable the strategy.**
5. If there's an exception or daemon crash, run the relevant
   `freeze_contingencies.md` §C1 step. Do NOT make a behaviour change.

## Weekly review (Friday EOD, 30 minutes)

### Friday format — `docs/freeze_week<N>_review.md`

The review is a table, not commentary. Numbers go on a page; you
stare at them. No decisions in Weeks 1 and 2.

| Strategy | Live N | Live PF | Live PF lower-CI | Live Long PnL | Live Short PnL | Battery PF (latest) | Live/Battery Δ |
|---|---:|---:|---:|---:|---:|---:|---:|
| breakout |  |  |  |  |  |  |  |
| (… every strategy …) |  |  |  |  |  |  |  |

**Plus** one row in `logs/drift/weekly_variance.csv` (machine-readable
trend, persisted across the freeze window):

```
week_ending,strategy,live_n,live_pf,battery_pf,delta_pct,within_band
2026-05-22,breakout,8,1.23,1.08,+14%,YES
2026-05-22,rsi_momentum,4,0.55,0.92,-40%,NO
...
```

`within_band` = `YES` if `|delta_pct| <= 20%`, `NO` otherwise.

### Hard rule for weekly reviews

- Week 1 (Fri May 22): **NO decisions.** 5 days of data is statistical
  noise. Operational issues (daemon down, longs not firing, scanner
  empty) are the only thing that can prompt action.
- Week 2 (Fri May 29): **Mid-freeze health check.** Use the 5-question
  template below. If all 5 are YES, stay the course. If #1 (longs ≥
  30 % of trades) is NO, investigate the long-entry path as a bug.
- Week 3 (Fri June 5): produce the pre-decision artefacts (per-strategy
  verdict table, battery agreement table, calibration drift log, ops
  reliability). Decision happens **June 8 Monday morning**, market
  closed, not Friday EOD.

### Week 2 mid-freeze 5-question health check

One sentence each, no more:

1. Is the system trading both directions? (Long N ≥ 30 % of total N
   counts as yes.)
2. Are any strategies at N ≥ 30 yet?
3. Are battery PF estimates within ±15 % of live PF for any strategy?
4. Is the per-trade slippage proxy (paper fill vs signal-time price)
   under 0.15 %?
5. Has the daemon stayed up > 95 % of market minutes this week?

Decision gates after the 5 answers — see `FREEZE_v2.1.md`
§Operator commitments and §Disagreement.

## EOD diagnostic — required outputs (2026-05-19 extensions)

The EOD diagnostic produces a markdown report at
`logs/diagnostics/eod_YYYY-MM-DD.md`. The 2026-05-19 commit added
the following stats sections in support of `freeze_contingencies.md` §C2.

### Per-strategy verdict table (existing, unchanged)

| Strategy | N | WR % | PF | R:R | BE WR % | Edge pp | Expectancy | Total PnL | Kelly | Verdict |

### NEW — PF excluding maximum-PnL trade per strategy

Detects "edge is one lucky trade" pattern.

| Strategy | N | PF | PF-excl-max | Δ |
|---|---:|---:|---:|---:|

Decision rule (`freeze_contingencies.md` §C2):
PF > 1.0 AND PF-excl-max < 0.8 → verdict downgraded to INSUFFICIENT.

### NEW — PF lower-95-CI (bootstrap)

The "PF" point estimate is uninformative on N < 30. The bootstrap
lower-CI is what an honest verdict uses.

| Strategy | N | PF (point) | PF lower-95-CI | PF upper-95-CI |
|---|---:|---:|---:|---:|

Method: 1000 resamples with replacement of per-trade PnL; compute PF
on each resample; report 2.5 % / 97.5 % percentiles. Implemented in
`packages/research/diagnostic.py` →`bootstrap_pf_ci()`.

### NEW — Per-supersector PF

Detects sector-concentration artefacts.

| Supersector | N | Wins | Losses | PF | PnL |
|---|---:|---:|---:|---:|---:|

Decision rule: if PF varies by > 50 % across supersectors with N ≥ 5
each, the edge is sector-dependent. Document but do not unfreeze.

### NEW — Entry-time histogram per strategy

Detects time-of-day artefacts.

| Strategy | 09:15–10:00 | 10:00–11:00 | 11:00–12:00 | 12:00–13:00 | 13:00–14:00 | 14:00–15:30 |
|---|---:|---:|---:|---:|---:|---:|

Decision rule: if > 70 % of a strategy's winning trades fire in a
single 90-minute window, the edge is time-dependent. Document, watch
during the freeze.

### NEW — Contaminated-days handling

Defined in `freeze_contingencies.md` §C8.

- `logs/contaminated_days.csv` schema: `date, vix, nifty_pct, reason`.
- The EOD diagnostic reports BOTH:
  - **Inclusive PF** — every closed trade in the window
  - **Exclusive PF** — same window minus contaminated days
- Phase A edge decision (June 8) uses the **exclusive** PF.
- Phase A risk-tolerance decision uses the **inclusive** PF.

## Heartbeat email (2026-05-19 addition)

Single-line confirmation that the trader is alive, sent at 09:10 IST
daily — before the 09:15 market open. The heartbeat is the operator's
silent-failure detector for §C1.b (alert pipeline broken).

### What it says

Subject: `[Trading Agent] Heartbeat YYYY-MM-DD`

Body (≈ 5 lines, markdown for the HTML renderer):

```
## Trading Agent Heartbeat

- **Daemon:** UP since YYYY-MM-DD HH:MM:SS IST
- **Uptime:** X h Y m
- **Last EOD:** Rs ±__  | N closed trades
- **Last audit checkpoint:** GREEN / AMBER / RED at HH:MM IST
- **Open positions:** N
- **Spool depth (failed alerts):** N
```

### What it means if it doesn't arrive

The contract: **if the 09:10 IST email does not arrive by 09:25 IST,
follow `freeze_contingencies.md` §C1.b**. Possible causes (ordered by
likelihood):

1. SMTP / Resend credential expired or rate-limited.
2. Daemon container down.
3. VM unreachable.
4. Cron / systemd timer didn't fire on the VM.

### Where it lives

- `tools/send_heartbeat.py` — the script. Reads the latest
  `data/trading_agent.db` + `logs/health.json` + latest audit
  checkpoint, composes a one-block markdown body, sends via the
  existing `AlertManager`.
- `tools/cloud/install_heartbeat_cron.sh` — installs a cron entry
  `10 9 * * 1-5 cd /home/ubuntu/trading-agent && python tools/send_heartbeat.py`.
- Failure mode: if the script itself crashes, it still emits a
  minimal "heartbeat failed: <reason>" email so the operator knows
  the daemon at least had a working alert path.

## Drift directory layout (2026-05-19 addition)

A new directory `logs/drift/` tracks the battery-vs-live agreement
trend. Schema:

```
logs/drift/
├── weekly_variance.csv          # one row per Friday per strategy
├── slippage_proxy_YYYY-MM-DD.csv # per-trade fill-vs-signal price delta
├── calibration_xgboost.csv      # per-trade predicted_proba vs realised outcome
└── README.md                    # this layout summary
```

`weekly_variance.csv` is produced by the Friday weekly review (manual
script for now; auto-emit by Friday June 5 if `tools/research/`
gets a `weekly_variance.py` added — that's freeze-safe observability).
